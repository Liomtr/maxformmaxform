目标是继续收口工作台和管理后台里已经有界面但后端能力不完整的模块，优先补齐以下几条链路：

- 部门管理从“可创建”推进到“可编辑、可删除”
- 成员导入从前端伪批量改成真实后端导入接口
- 工作台中的成员管理动作补齐
- 文件夹、消息、审计日志从占位接口改成真实接口
- 问卷回收站从纯前端界面改成真实可用的软删除/恢复/清空流程
- 为部门删除与成员导入补后端测试
- 将当天开发记录整理成一份完整日志

## 完成内容

### 1. 收口工作台中的组织管理

修改文件：

- `frontend/src/views/UserDashboard.vue`
- `backend/src/models/Dept.js`
- `backend/src/routes/depts.js`

主要改动：

- 工作台中的“团队/部门”区域不再只是说明性占位，而是接入真实部门与成员数据。
- 增加了部门创建、部门编辑、部门删除入口。
- 部门弹窗改成创建/编辑复用模式，减少重复 UI。
- 删除部门前增加后端保护逻辑：
  - 如果部门下还有子部门，直接拒绝删除
  - 如果部门下还有成员，先清空这些成员的 `dept_id`
  - 删除成功后返回本次被解除部门归属的成员数量

结果：

- 部门管理形成了最小闭环，已经不再是“只能新增、不能维护”的半成品状态。

### 2. 收口成员管理与成员导入

修改文件：

- `frontend/src/views/UserDashboard.vue`
- `frontend/src/api/userAdmin.ts`
- `backend/src/routes/users.js`

主要改动：

- 工作台中补齐成员编辑、启用/禁用、重置密码、删除等管理动作。
- 成员导入不再在前端逐条伪造统计结果，而是接入真实后端接口 `POST /api/users/import`。
- 后端导入逻辑对以下情况做了显式统计：
  - 用户名为空
  - 密码为空
  - 本次导入数据内用户名重复
  - 数据库中用户已存在
- 导入接口返回统一结构：
  - `created`
  - `skipped`
  - `errors`

结果：

- 成员导入与成员管理能力已经从“界面存在但能力不完整”推进到前后端一致可用。

结果：

- 工作台中的回收站区域已经从纯展示界面变成真实可操作流程。
