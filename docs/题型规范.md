# 问卷题型规范

基于源码状态更新时间：2026-03-25

本文档以当前仓库代码为准，用于描述题型系统的真实实现方式。

若仓库中其它文档仍提到 `MongoDB`、`Question.js`、`Question._id`、`ClickHouse` 异步统计等历史方案，以本文和当前源码为准。

## 一、规范目标

当前题型规范的目标不是描述“未来理想态”，而是统一以下事实：

- 后端如何保存题目
- 前端如何映射 legacy 题型
- 哪些题型已进入真实支持范围
- 提交答案时后端如何做标准化与校验
- 结果统计页当前能输出哪些题型统计结构

## 二、当前架构事实

### 2.1 数据存储

当前项目主线采用 `MySQL + JSON 字段` 方案：

- 问卷定义存放在 `surveys` 表
- 题目列表整体存放于 `surveys.questions`
- 答卷存放在 `answers` 表
- 答案明细整体存放于 `answers.answers_data`

对应源码：

- `backend/src/models/Survey.js`
- `backend/src/models/Answer.js`

当前不存在以下历史结构：

- 独立的 `backend/src/models/Question.js`
- `MongoDB Question._id` 作为题目主键
- 提交时将题号转换为 `Question._id`
- ClickHouse 作为当前正式统计链路

### 2.2 题型权威来源

当前“后端权威题型”并不是数据库 enum 字段，而是由以下逻辑共同确定：

- `backend/src/utils/questionSchema.js`
- `frontend/src/utils/questionTypeRegistry.ts`

后端支持识别的服务端题型为：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `date`
- `slider`
- `ranking`
- `upload`
- `rating`
- `scale`
- `matrix`

其中真正已形成较完整闭环的，以当前前后端联动结果为准，见下文。

## 三、题型模型

### 3.1 问题对象的当前结构

`surveys.questions[]` 中的单题对象当前采用宽松 JSON 结构，典型字段如下：

```jsonc
{
  "id": "frontend-generated-id",
  "type": "radio",         // 当前保存到后端的服务端题型
  "uiType": 7,             // 可选：前端 legacy 题型编号，用于还原编辑器语义
  "title": "题目标题",
  "titleHtml": "<p>富文本标题</p>",
  "description": "题目说明",
  "required": false,
  "order": 1,
  "options": [
    {
      "label": "选项1",
      "value": "1",
      "order": 1,
      "rich": false,
      "desc": "",
      "hidden": false,
      "exclusive": false,
      "defaultSelected": false,
      "visibleWhen": null,
      "fillEnabled": false,
      "fillRequired": false,
      "fillPlaceholder": ""
    }
  ],
  "optionOrder": "none",
  "optionGroups": [],
  "logic": {},
  "jumpLogic": {},
  "validation": {},
  "upload": {
    "maxFiles": 1,
    "maxSizeMb": 5,
    "accept": ".jpg,.jpeg,.png,.gif,.webp,.pdf,.docx,.xlsx"
  },
  "hideSystemNumber": false,
  "quotasEnabled": false,
  "quotaMode": "explicit",
  "quotaShowRemaining": false,
  "quotaFullText": "",
  "autoSelectOnAppear": false
}
```

说明：

- `type` 是当前保存到后端的服务端题型。
- `uiType` 是前端 legacy 编码的补充信息，用于区分“同样存成 `radio`，但编辑器语义其实是下拉题”等场景。
- 后端不会强依赖固定 schema，但会在保存、发布、提交阶段做最小标准化与校验。

### 3.2 选项题的规范

当前被视为“选项题”的后端类型为：

- `radio`
- `checkbox`
- `ranking`

后端会对这些题型的 `options` 做标准化：

- 统一补齐 `label`
- 统一补齐 `value`
- 统一补齐 `order`

## 四、前端题型体系

### 4.1 legacy 题型编码

前端编辑器仍保留 legacy 数字题型，用于：

- 左侧题型面板
- 编辑器内部切换
- 兼容历史草稿
- 通过 `uiType` 还原具体 UI 语义

当前注册表位于：

- `frontend/src/utils/questionTypeRegistry.ts`

### 4.2 legacy 到服务端题型映射

当前映射如下：

| legacy | 名称 | 保存为服务端题型 |
|---|---|---|
| 1 | 填空题 | `input` |
| 2 | 简答题 | `textarea` |
| 3 | 单选题 | `radio` |
| 4 | 多选题 | `checkbox` |
| 7 | 下拉题 | `radio` |
| 8 | 滑动条题 | `slider` |
| 11 | 排序题 | `ranking` |
| 13 | 文件上传 | `upload` |
| 14 | 日期 | `date` |

说明：

- 下拉题当前并不是独立持久化类型，而是通过 `type='radio' + uiType=7` 表达。
- 段落说明当前通过 `uiType=18` 表达，通常仍落在普通文本类型体系中，由前端渲染层识别为说明块。

### 4.3 当前已实现的 legacy 题型

当前注册表中已标记为实现完成的 legacy 题型为：

- `1` 填空题
- `2` 简答题
- `3` 单选题
- `4` 多选题
- `7` 下拉题
- `8` 滑动条题
- `11` 排序题
- `13` 文件上传
- `14` 日期
- `18` 段落说明

其它编辑器中展示的题型，如矩阵、评分、NPS、比重题、AI 题型等，当前仍不应视为已闭环题型。

## 五、保存与读取规范

### 5.1 前端保存到后端

前端保存问卷时：

- 将编辑器内部 `type` 视为 legacy 编码
- 通过 `mapLegacyTypeToServer()` 转换为服务端题型
- 同时保留 `uiType`

保存后典型结果如下：

```jsonc
{
  "type": "radio",
  "uiType": 7
}
```

这表示：

- 持久化上按 `radio` 处理
- 编辑器语义上仍可还原为“下拉题”

### 5.2 后端标准化保存

后端在创建和更新问卷时，会通过 `normalizeSurveyQuestions()` 做标准化：

- 标准化题型
- 统一题目顺序 `order`
- 标准化选项
- 标准化上传题配置

其中：

- 非法或未知题型会回退为 `input`
- `dropdown` 会被收敛为 `radio`

### 5.3 后端返回前端

前端读取问卷后，会通过：

- `mapServerTypeToLegacy(type, uiType)`

将后端题型还原为编辑器内部 legacy 题型。

当前优先使用 `uiType` 还原；若缺失，再按 `type` 回推 legacy 值。

这使得下拉题、段落说明等“共享服务端题型但 UI 语义不同”的场景可以被正确还原。

## 六、填写页渲染规范

当前填写端真实支持的专属渲染如下：

| 服务端题型 | 渲染方式 |
|---|---|
| `input` | 单行输入框 |
| `textarea` | 多行输入框 |
| `radio` | 单选框组 |
| `checkbox` | 多选框组 |
| `ranking` | 上移/下移排序交互 |
| `upload` | 文件上传控件 |
| `date` | 日期输入控件 |
| `slider` | 滑动条控件 |

通过 `getQuestionRenderType()` 额外识别的 UI 类型：

- `uiType=7` 渲染为 `dropdown`
- `uiType=18` 渲染为 `stage_explain`

因此当前填写页可视为已支持的真实题型集合为：

- 填空
- 简答
- 单选
- 多选
- 下拉
- 排序
- 文件上传
- 日期
- 滑动条
- 段落说明

以下服务端题型虽然在后端支持识别，但当前填写端尚未形成完整专属交互：

- `rating`
- `scale`
- `matrix`

## 七、提交流程与答案结构

### 7.1 当前提交格式

前端提交答卷时，当前发送结构为：

```jsonc
{
  "answers": [
    {
      "questionId": 1,
      "value": "A"
    }
  ],
  "duration": 35,
  "clientSubmissionToken": "..."
}
```

说明：

- `questionId` 当前是题目顺序号（1-based）。
- 当前并不会映射为独立题目表主键。
- 上传题会额外使用 `clientSubmissionToken` / `submissionToken` 完成临时文件归属与提交绑定。

### 7.2 后端标准化后的答案结构

后端接收提交后，会通过 `validateSubmissionAnswers()` 标准化为：

```jsonc
{
  "questionId": 1,
  "questionType": "radio",
  "value": "1"
}
```

因此，当前 `answers.answers_data[]` 的事实结构是：

```jsonc
[
  {
    "questionId": 1,
    "questionType": "radio",
    "value": "1"
  },
  {
    "questionId": 2,
    "questionType": "checkbox",
    "value": ["1", "3"]
  }
]
```

对上传题，提交后会变成：

```jsonc
{
  "questionId": 5,
  "questionType": "upload",
  "value": [
    {
      "id": 101,
      "name": "a.pdf",
      "url": "/uploads/...",
      "size": 123,
      "type": "application/pdf"
    }
  ]
}
```

### 7.3 当前后端提交校验

后端当前已实现的提交校验包括：

- `radio` 的选项合法性校验
- `checkbox` 的数组结构与选项合法性校验
- `ranking` 的数组结构与选项合法性校验
- `upload` 的数组结构、文件数量、大小、类型校验
- `slider` 的数值校验
- `required` 必填校验

这部分能力由 `backend/src/utils/questionSchema.js` 提供。

## 八、上传题规范

上传题是当前题型系统里最特殊的一类，采用“两阶段提交”：

### 8.1 上传阶段

前端先调用：

- `POST /api/surveys/:id/uploads`

请求中包含：

- 文件本体
- `questionId`
- `submissionToken`

后端会：

- 校验该问卷是否存在上传题
- 校验目标题目是否真的是上传题
- 校验数量、大小、扩展名/MIME
- 将文件记录为待提交状态
- 返回 `uploadToken`

### 8.2 答卷提交阶段

正式提交答卷时，后端会：

- 根据 `submissionToken` 找到待提交文件
- 校验文件与题目匹配关系
- 将文件转为最终答案值
- 将文件绑定到本次答卷
- 清理待提交状态

因此，上传题的事实标准不是简单字符串 URL，而是“文件对象数组”。

## 九、发布校验规范

当前发布问卷时，后端会通过 `validateSurveyQuestions()` 执行最小校验：

- 问卷至少一题
- 每题必须有标题
- `radio` / `checkbox` / `ranking` 必须至少两个选项

这意味着：

- 当前发布校验已不只是标题检查
- 但仍属于“最小校验”，不是完整 schema 校验

尚未形成完整发布校验的题型包括：

- `rating`
- `scale`
- `matrix`

## 十、统计输出规范

当前结果接口：

- `GET /api/surveys/:id/results`

结果统计并非走 ClickHouse，而是直接基于 `answers.answers_data` 聚合。

当前 `buildQuestionStats()` 支持输出以下统计：

### 10.1 选择题

适用：

- `radio`
- `checkbox`

输出：

- `options[]`
- 每个选项的 `count`
- 每个选项的 `percentage`

### 10.2 文本题

适用：

- `input`
- `textarea`

输出：

- `sampleAnswers`
- `totalAnswers`

### 10.3 滑动条题

适用：

- `slider`

输出：

- `avgValue`
- `minValue`
- `maxValue`

### 10.4 排序题

适用：

- `ranking`

输出：

- `options[]`
- `count`
- `percentage`
- `avgRank`

### 10.5 上传题

适用：

- `upload`

输出：

- `totalFiles`
- `sampleFiles`

### 10.6 日期题

适用：

- `date`

输出：

- `sampleAnswers`
- `earliestDate`
- `latestDate`

### 10.7 其它题型

当前下列题型还没有完整统计适配：

- `rating`
- `scale`
- `matrix`

## 十一、当前建议的题型状态划分

### 11.1 已闭环题型

当前可视为已具备“编辑、保存、预览、填写、提交、统计”主链路闭环的题型：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `dropdown`（通过 `radio + uiType=7`）
- `slider`
- `ranking`
- `upload`
- `date`
- `stage_explain`（说明型，不参与答题统计）

### 11.2 半接通题型

当前服务端已识别，但尚未形成完整双端专属能力：

- `rating`
- `scale`
- `matrix`

### 11.3 占位题型

编辑器面板中仍可能出现但不应视为正式支持的题型，包括但不限于：

- 各类矩阵扩展
- NPS / 评分矩阵 / 比重题
- AI 题型
- 录音 / OCR / 视频等扩展型题目

## 十二、新增题型的正确流程

后续如需新增题型，建议按以下顺序推进：

1. 在 `frontend/src/utils/questionTypeRegistry.ts` 注册题型映射与标签。
2. 在 `backend/src/utils/questionSchema.js` 中补充：
   - 类型识别
   - 问题标准化
   - 发布校验
   - 提交校验
3. 在填写端补专属渲染：
   - `FillSurveyPage.vue`
   - `FillSurveyMobilePage.vue`
4. 在预览层补齐还原与渲染：
   - `surveyMappers.ts`
   - `usePreviewSurveyMapping.ts`
5. 在结果页补统计适配：
   - `backend/src/routes/surveys.js`
   - `frontend/src/views/survey/SurveyAnswersPanel.vue`
6. 同步更新本文档。

新增题型达到“正式支持”的最低标准应为：

- 编辑器可创建
- 保存后可稳定还原
- 预览正确
- 填写页有专属交互或明确兼容策略
- 提交有题型校验
- 结果页有基础统计或明确标注不支持统计
