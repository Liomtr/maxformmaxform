# 项目功能清单与风险分析

基于源码分析日期：2026-03-21

本文结论以当前仓库源码为准，不以历史文档为准。当前项目可以定位为“前后端分离的企业级问卷/表单平台”，但其完成度呈现明显分层：核心问卷流程已有基础闭环，企业化扩展能力和部分后台能力仍存在占位、接口未对齐、权限边界不足等问题。

## 1. 项目当前定位

- 前端是基于 Vue 3 的可视化问卷平台，覆盖首页、登录、用户工作台、问卷编辑器、填写页、结果页、管理后台。
- 后端是基于 Express 的 REST API 服务，负责认证、问卷、答卷、用户、角色、部门、文件等能力。
- 数据层实际使用 MySQL + Knex。
- 项目存在较明显的历史演进痕迹，旧文档中出现的 MongoDB、ClickHouse、适配器模式、内存版服务，与当前主干实现并不完全一致。

## 3. 当前缺陷与风险清单

以下按优先级分类。`P0` 表示会直接阻断主流程或启动，`P1` 表示高风险上线问题，`P2` 表示中期稳定性与可维护性问题，`P3` 表示低优先级但值得治理的问题。

### 3.1 P0 关键阻断问题

#### P0-1 问卷核心模型缺失，后端主流程存在直接失效风险

证据：

- `backend/src/routes/surveys.js` 引用了 `../models/Survey.js`
- 当前 `backend/src/models` 目录中不存在 `Survey.js`

影响：

- 问卷列表、创建、更新、发布、关闭、提交、结果查询都依赖该模型
- 如果仓库当前状态就是实际运行状态，则后端启动或相关接口调用会直接失败

结论：

- 这是当前最优先需要确认和修复的问题

### 3.2 P1 高风险问题

#### P1-1 问卷权限边界不足，存在横向越权风险

证据：

- `backend/src/routes/surveys.js` 中，更新、删除、发布、关闭接口只有 `authRequired`
- 未校验问卷是否属于当前用户，也未限制为管理员

影响：

- 任意已登录用户只要知道问卷 ID，就可能修改、删除、发布、关闭他人的问卷

#### P1-2 答卷读取与导出权限不足，存在数据泄露风险

证据：

- `backend/src/routes/answers.js` 中的列表、详情、统计、删除、导出仅要求登录
- 未验证该答卷或问卷是否属于当前用户

影响：

- 任意登录用户可能查看、导出、删除其他人的答卷数据

#### P1-3 文件管理权限不足，存在越权读取和删除风险

证据：

- `backend/src/routes/files.js` 的列表接口对所有登录用户开放
- 删除接口未校验文件归属或管理员权限

影响：

- 任意登录用户可浏览全量上传文件
- 任意登录用户可能删除他人上传的文件

#### P1-4 存在匿名图片上传入口，容易被滥用

证据：

- `backend/src/routes/files.js` 中 `/upload/image` 未加 `authRequired`

影响：

- 外部匿名请求可直接向服务器写入文件
- 容易导致存储滥用、恶意内容上传、资源占用异常

#### P1-5 未发布问卷可被直接读取，草稿有泄露风险

证据：

- `backend/src/routes/surveys.js` 中 `GET /:id` 是公开接口
- 未校验问卷状态是否为 `published`

影响：

- 任何人只要知道问卷 ID，就可能直接读取草稿或未发布问卷内容

#### P1-6 分享机制未真正收口，存在可枚举性和字段不一致问题

证据：

- 后端表结构有 `share_code`
- 前端工作台与填写页多数场景仍使用 `/s/:id`
- 前端多处使用 `shareId || id` 兼容逻辑

影响：

- 分享入口可能退化为直接暴露自增 ID
- 问卷公开访问路径更容易被枚举
- 前后端字段命名不统一增加维护成本

#### P1-7 前后端接口不一致，后台能力无法稳定闭环

典型证据：

- `frontend/src/views/admin/Surveys.vue` 调用 `/surveys/:id/audit`，后端不存在该接口
- 同一页面按用户名调用 `/users/${username}`，但后端用户详情路由设计为按 ID 查询
- 同一页面调用 `listSurveys({ createdBy: username })`，但前端 API 类型和后端接口都未支持该查询参数
- `frontend/src/api/userAdmin.ts` 中存在用户导入、导出、岗位分配等接口封装，后端未完整提供

影响：

- 管理后台部分页面只能作为展示原型，不能视为真实可用能力
- 开发联调成本高，问题定位困难

#### P1-8 文档与实现分叉严重，容易造成错误认知

证据：

- 历史文档提到 MongoDB、ClickHouse、适配器层、内存版服务
- 当前主干实现实际是 MySQL + Knex + Express

影响：

- 新成员会被错误文档带偏
- 部署、排障、二次开发容易基于错误假设开展

#### P1-9 问卷设置字段大量存在，但后端未真正执行

证据：

- 前端和类型定义中有 `allowMultipleSubmissions`、`endTime`、`submitOnce`、`examMode`、`timeLimit`、`randomOrder`
- 后端问卷提交和读取逻辑中未看到成体系的强制执行

影响：

- 产品上看似支持复杂控制，实际上提交规则可能不生效
- 容易引发业务侧误用或投诉

### 3.3 P2 中风险问题

#### P2-1 数据库迁移策略过于原始，后续演进困难

证据：

- `backend/src/db/migrate.js` 以 `users` 表是否存在作为整体迁移完成判断
- 一旦初始化完成，后续新增字段或新表无法通过标准迁移自动演进

影响：

- 版本升级困难
- 线上环境难以平滑变更 schema

#### P2-2 `initAdmin.js` 仍是旧架构脚本，与当前实现不一致

证据：

- 该脚本仍引用旧的 `database` 适配器与 Mongo 风格模型
- 与当前 `Knex + MySQL` 主线不一致

影响：

- 运维或开发人员可能误用该脚本
- 管理员初始化流程存在误导

#### P2-3 占位接口返回“成功空数据”，会掩盖真实未实现状态

证据：

- `backend/app.js` 中 `messages/folders/positions/audits` 直接返回空数组或空分页

影响：

- 前端页面可能看起来“没有数据”，而不是“功能未接入”
- 联调和验收时容易误判完成度

#### P2-4 后端对答卷数据结构校验偏弱

证据：

- `backend/src/routes/surveys.js` 提交答卷时仅检查问卷存在且已发布
- `answers_data` 基本按原样写入数据库

影响：

- 容易写入脏数据
- 难以保证答卷内容与题目结构一致

#### P2-5 健康检查过于轻量，无法反映关键依赖状态

证据：

- `GET /health` 只返回状态和时间

影响：

- 运维难以及时判断数据库连接、磁盘写入、上传目录状态等关键依赖是否正常

#### P2-6 缺少系统化测试

现状：

- 仓库有若干脚本文件，但未见成体系的单元测试、集成测试或 E2E 测试框架接入

影响：

- 当前这种“前端很大、接口很多、状态字段很多”的项目，缺少测试会放大回归风险

### 3.4 P3 低优先级问题

#### P3-1 仓库存在多套历史文件和命名残留，理解成本偏高

典型表现：

- `frontend/frontend/package.json` 这类嵌套目录残留
- `server-cjs.js`、旧文档、旧启动方式与主线并存

影响：

- 增加项目理解和维护成本

#### P3-2 前端存在较多兼容字段映射逻辑，说明领域模型尚未稳定

典型表现：

- `responseCount/answerCount/submitCount`
- `shareId/share_code/id`
- `createdAt/updatedAt/publishedAt` 与后端蛇形命名并存

影响：

- 代码可读性下降
- 容易产生隐蔽字段兼容 bug

## 4. 总结结论

从源码角度看，这个项目已经不是单页 Demo，而是一个具备完整产品野心的问卷平台。它的优势在于：

- 前端产品面较宽
- 编辑器交互能力丰富
- 基础后台能力和组织模型已有雏形
- 问卷、答卷、用户、角色、部门、文件这些核心实体都已经进入数据库设计

但当前不适合直接按“企业级平台已完成”来判断，原因主要有四类：

- 核心问卷模型实现存在缺口
- 权限边界不足，存在真实的越权与数据泄露风险
- 前后端接口未完全对齐
- 文档、旧脚本、历史方案与现代码分叉较大

如果要进入下一阶段，建议优先顺序如下：

1. 先补齐 `Survey` 模型并确认问卷主链路可启动、可跑通。
2. 立即修补问卷、答卷、文件三个模块的权限校验。
3. 清理前后端接口不一致项，先让后台管理页只保留已闭环能力。
4. 统一分享字段、时间字段、计数字段命名，收敛领域模型。
5. 清理旧文档和旧脚本，重新定义“当前唯一可信启动方式”。
