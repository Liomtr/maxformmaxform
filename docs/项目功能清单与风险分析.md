# 项目功能清单与风险分析

基于源码状态更新时间：2026-03-23

本文结论以当前仓库源码为准，不以历史文档、旧方案草稿或早期演示页面为准。当前项目已经具备“前后端分离问卷平台”的主链路闭环，但企业化后台能力仍处于“基础可用、持续收口”的阶段。

## 1. 当前产品定位

- 前端：基于 Vue 3 + Element Plus 的问卷平台，覆盖首页、登录、用户工作台、问卷编辑器、填写页、结果页和后台管理页。
- 后端：基于 Express + Knex + MySQL 的 REST API，负责认证、问卷、答卷、用户、角色、部门、文件等核心能力。
- 数据层：当前主线以 MySQL 为准，不再以历史文档中的 MongoDB / ClickHouse 方案作为当前实现依据。

## 2. 模块级功能清单

### 2.1 认证与会话

已实现：

- 用户注册
- 用户登录
- JWT 签发与校验
- 获取当前用户信息
- 路由级登录态守卫
- 基于角色码的管理员权限判断

现状判断：

- 该模块已具备稳定可用基础，是当前较完整的业务底座之一。

### 2.2 问卷核心生命周期

已实现：

- 创建问卷
- 查询问卷列表
- 查询单个问卷
- 更新问卷
- 删除问卷
- 发布问卷
- 关闭问卷
- 通过公开入口读取已发布问卷
- 提交答卷
- 查询基础结果统计

已收口的关键规则：

- 草稿 / 关闭状态问卷不再允许被公开访问
- 问卷管理操作受创建者 / 管理员权限约束
- 答卷提交支持截止时间与重复提交限制

现状判断：

- 核心问卷链路已经可运行，可支撑基础创建、发布、填写、查看结果。
- 更复杂的高级规则仍有继续深化空间，但主链路不再是“原型状态”。

### 2.3 问卷编辑器

已实现：

- 标题、描述、题目增删改查
- 题目排序、复制、预览
- 多类基础题型编辑
- 题目基础属性与选项配置
- 问卷保存与发布
- 部分逻辑跳题与显示控制
- `settings` 持久化基础打通

仍需继续收口：

- 大量高级题型仍属于 UI 先行、后端能力未完全对齐
- 编辑器内部仍存在较多 legacy 类型映射逻辑

现状判断：

- 编辑器能力明显强于后台落地能力，是当前功能最丰富、但也最需要持续治理的模块之一。

### 2.4 填写端

已实现：

- PC 填写页
- 移动端填写页
- 基础校验
- 基础显示/隐藏与跳题
- 成功页跳转
- 公开分享入口

已收口能力：

- 截止时间限制
- 重复提交限制
- 已发布 / 未发布边界

现状判断：

- 填写端已能支撑真实问卷收集。
- 复杂逻辑题和更强后端校验仍有提升空间。

### 2.5 答卷管理与导出

已实现：

- 按问卷查询答卷
- 查询答卷总数
- 查看答卷详情
- 删除答卷
- 导出 Excel
- 结果页展示总提交数与最近提交时间

已收口能力：

- 答卷读取、导出、删除已纳入权限控制

现状判断：

- 适合基础运营与管理使用。
- 尚未形成成熟的数据分析产品能力。

### 2.6 用户、角色、部门管理

已实现：

- 用户列表
- 用户创建
- 用户更新
- 用户启用 / 禁用
- 用户密码重置
- 用户删除
- 角色列表、创建、更新、删除
- 部门列表、树形结构、创建、更新、删除

2026-03-23 进一步收口：

- 新增真实的 `POST /api/users/import` 批量导入接口
- 用户工作台内可直接导入成员
- 用户工作台内可直接编辑成员、启停成员、重置密码、删除成员
- 用户工作台内可直接编辑部门、删除部门
- 删除部门时：
  - 若存在子部门，后端会拒绝删除
  - 若存在成员，后端会先清空成员部门归属，再删除部门

现状判断：

- 组织管理已从“后台原型页”推进到“工作台内最小可用闭环”。
- 仍缺少更重型的组织能力，例如审批、邀请码、组织级策略继承、岗位体系等。

### 2.7 文件上传与富文本资源

已实现：

- 普通文件上传
- 图片上传
- 文件列表
- 文件删除
- 富文本编辑器接入上传资源

已收口能力：

- 文件列表与删除已加归属校验
- 匿名图片上传已关闭

现状判断：

- 该模块已具备基础支撑作用，但在更细粒度的权限和治理上仍有扩展空间。

### 2.8 用户工作台与后台界面

已实现：

- 用户工作台
- 问卷列表
- 回收站
- 文件夹
- 组织管理入口
- 后台成员、角色、部门、问卷等页面

2026-03-23 当前真实状态：

- 工作台中的“团队 / 部门”区域不再是 `TODO` 占位
- 已具备：
  - 协作概览
  - 部门列表
  - 最近成员列表
  - 部门创建 / 编辑 / 删除
  - 成员导入
  - 成员编辑 / 启停 / 重置密码 / 删除

仍属于占位或半成品的部分：

- 独立 Team 模型
- 邀请码 / 加入团队流程
- 通讯录
- 模板中心深度能力
- 审批流
- 审计中心
- 独立消息中心产品化能力

现状判断：

- 工作台已开始从展示型页面转向可操作页面。
- 但企业协作能力仍未完全平台化。

### 2.9 占位接口与规划模块

当前仍明显偏占位的模块：

- `messages`
- `folders`
- `positions`
- `audits`

现状判断：

- 这些模块目前更接近“接口壳”或“演示入口”。
- 不应在功能清单中被等同视作已完整交付能力。

## 3. 当前风险清单

### 3.1 P1 高风险问题

#### P1-1 高级编辑能力与后端落地能力仍不完全对齐

表现：

- 前端编辑器支持的高级题型和规则明显多于后端真实持久化与校验能力。

影响：

- 用户看到的配置能力可能大于系统真正可执行的规则能力。

#### P1-2 协作模型仍由“部门”代偿，尚未形成独立 Team 领域模型

表现：

- 当前“创建协作组”本质上落到部门数据。
- 尚无邀请码、加入流程、协作审批与团队级权限边界。

影响：

- 若后续继续扩张协作需求，部门模型会逐渐承担超出其语义边界的职责。

#### P1-3 批量导入虽已成真实接口，但仍是逐条处理

表现：

- `/api/users/import` 已真实存在，但当前仍是接口内逐条校验与创建。

影响：

- 大批量导入时，吞吐与错误处理体验仍有限。

### 3.2 P2 中风险问题

#### P2-1 数据库迁移策略仍较粗糙

表现：

- 当前迁移脚本更接近初始化脚本，而不是成熟的版本化迁移体系。

影响：

- 后续 schema 演进成本较高。

#### P2-2 前端 chunk 体积仍然偏大

表现：

- `vite build` 持续提示大 chunk 警告。

影响：

- 首屏加载和后续性能优化仍有明确空间。

#### P2-3 部分模块仍以占位接口支撑

表现：

- 消息、审计、岗位、文件夹等模块仍缺乏真实后端闭环。

影响：

- 页面上虽有入口，但不能等同视作真实产品能力。

### 3.3 P3 低优先级但应持续治理的问题

#### P3-1 领域字段仍存在较多兼容映射

表现：

- `responseCount / answerCount / submitCount`
- `shareId / share_code / id`
- 多套时间字段兼容逻辑

影响：

- 可维护性下降，容易产生隐蔽兼容 bug。

#### P3-2 历史文档与旧方案残留较多

表现：

- 仓库中仍存在早期方案文档与历史设计痕迹。

影响：

- 新成员容易基于旧认知继续开发。

## 4. 当前总体判断

当前项目已经不是单页演示或纯 UI 原型，而是一个具备真实主链路的问卷平台：

- 问卷创建、发布、填写、结果查看已闭环
- 认证、权限、答卷、文件、组织管理已有可运行基础
- 工作台中的组织能力在 2026-03-23 已明显前移

但它还不能被判断为“企业级平台已全部完成”，主要差距集中在：

- 高级编辑能力与后端规则执行的一致性
- 独立团队协作模型
- 占位接口的继续清理
- 数据分析深度
- 性能与迁移治理

## 5. 建议的后续优先级

1. 为部门 / 成员导入补正式后端测试，确认删除与批量导入边界。
2. 决定“团队”是否需要独立后端模型；若需要，先定义 Team、Member、Invite 的领域边界。
3. 继续收口占位模块，优先处理 `folders / messages / audits`。
4. 逐步减少前端 legacy 映射字段，统一领域模型命名。
5. 处理前端 chunk 拆分和构建体积问题。
