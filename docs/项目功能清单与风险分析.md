# 项目功能清单与风险分析

基于源码状态更新时间：2026-03-25

本文仅以 `maxformmaxform` 当前源码为准，不以历史方案、旧文档或过期演示界面为准。

## 1. 当前产品定位

当前项目已经不是单纯的前端原型，而是具备真实主链路的前后端分离问卷平台：

- 前端：Vue 3 + TypeScript + Vite 的问卷产品前台与后台
- 后端：Express + Knex + MySQL 的 REST API
- 核心闭环：创建问卷、发布问卷、公开填写、提交答卷、查看结果、导出答卷

但它还不是一个完全收口的企业级协作平台，尤其在 Team 领域、图表分析和后台壳层模块方面仍存在明显未完成部分。

## 2. 当前功能清单

### 2.1 认证与会话

已实现：

- 用户注册
- 用户登录
- JWT 鉴权
- 获取当前登录用户
- 基于角色码的管理员权限控制

当前判断：

- 这是比较稳定的底座模块之一。

### 2.2 问卷生命周期

已实现：

- 创建问卷
- 查询问卷列表
- 查询单个问卷
- 编辑问卷
- 发布问卷
- 关闭问卷
- 软删除到回收站
- 回收站恢复
- 回收站彻底删除
- 清空回收站
- 公开分享访问
- 文件夹归类与移动

已收口规则：

- 只有发布状态才允许公开填写
- 支持截止时间控制
- 支持重复提交限制
- 问卷管理受“管理员或创建者”约束

当前判断：

- 核心问卷主链路已真实可用。

### 2.3 题型编辑、填写与统计

已形成真实闭环的题型：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `dropdown`
- `slider`
- `ranking`
- `upload`
- `date`
- `rating`
- `scale`
- `matrix`
- `stage_explain`（内容块）

已实现能力：

- 编辑器配置
- 保存到问卷 JSON
- 公开填写页渲染
- 后端提交校验
- 结果页题目级统计

当前判断：

- 题型能力已经明显超过“基础问卷 Demo”阶段。

### 2.4 答卷管理与导出

已实现：

- 按问卷分页查询答卷
- 查看单份答卷详情
- 统计答卷数量
- 批量删除答卷
- 导出 Excel

结果接口当前会返回：

- 总提交数
- 今日新增
- 完成数 / 未完成数
- 完成率
- 平均时长
- 系统环境统计
- 题目级统计

当前判断：

- 已经能支撑基本运营、复核和导出。

### 2.5 文件上传

已实现：

- 登录态普通文件上传
- 登录态图片上传
- 问卷公开附件上传
- 上传后在答卷提交阶段绑定到答案
- 上传文件删除

已收口规则：

- 上传题支持数量、大小、类型限制
- 文件与问卷、题号、提交会话绑定校验
- 待提交的临时文件支持过期清理

当前判断：

- 上传能力已是真实后端闭环，不再是单纯前端假上传。

### 2.6 用户、角色、部门管理

已实现：

- 用户列表、创建、编辑、删除
- 用户启用/禁用
- 用户密码重置
- 成员批量导入
- 角色增删改查
- 部门增删改查
- 部门树读取

已收口规则：

- 用户管理仅管理员可操作
- 删除部门前会拦截存在子部门的情况
- 删除部门时若仍有成员，会先清空成员部门归属

当前判断：

- 已经具备最小可用组织管理能力。

### 2.7 文件夹、消息与审计

已实现：

- 文件夹创建、重命名、层级管理、删除
- 删除文件夹前检查子文件夹
- 删除文件夹时把问卷移动回根目录
- 消息列表、消息已读
- 审计日志分页查询
- 问卷、用户、部门、文件夹等操作自动记录审计与消息

当前判断：

- 这是当前后台中比较扎实的一组辅助能力。

### 2.8 工作台与管理后台页面

已接入真实数据的部分：

- 用户工作台问卷列表
- 回收站
- 文件夹视图
- 团队/部门面板中的部门列表与最近成员
- 消息抽屉
- 结果页真实题目统计

仍明显带有占位属性的部分：

- 协作空间中的 Team 概念仍主要是说明性 UI
- 结果页的趋势图、地域分布、系统图主体仍是占位图
- 高级工具、SPSS 导出仍是占位按钮
- 附件批量下载仍提示“待后端接口支持”

### 2.9 壳层或预留接口

当前仍应视为未完成能力，而不是已交付功能：

- `GET /api/positions` 目前固定返回空数组
- 独立 Team / Invite / Approval 模型未落地
- 部分独立管理页仍偏演示入口或轻量封装

## 3. 当前主要风险

### 3.1 P1 风险

#### P1-1 题型体系仍存在 legacy 编码与 server type 并行维护

表现：

- 共享注册表虽然已经建立，但编辑器、填写页、映射器、统计面板仍保留不少分散判断
- 既有 `uiType`，又有 server `type`，还夹杂少量历史别名

影响：

- 后续继续扩展题型时，容易出现某个环节忘记补齐的回归问题

#### P1-2 协作语义仍由“部门”代管，独立 Team 领域尚未建立

表现：

- 用户工作台已经出现“协作”“团队/部门”等产品语义
- 但真实后端领域模型仍主要是 `dept + user + role`

影响：

- 如果继续扩展协作能力，部门模型会承担超出其语义边界的职责

#### P1-3 结果分析模块处于“数据真实、展示未收口”的中间态

表现：

- 题目统计数据是真实接口
- 趋势图、地域图、系统图和高级分析工具仍是 UI 预留

影响：

- 容易造成“分析能力已完成”的错误预期

### 3.2 P2 风险

#### P2-1 数据库迁移仍是“启动时补表/补字段”模式

表现：

- 当前 `migrate.js` 更像 schema ensure 脚本，而不是严格版本化迁移体系

影响：

- 未来字段演进、灰度升级、多人并行维护时，变更可追踪性和回滚能力偏弱

#### P2-2 成员导入仍为逐条处理

表现：

- `/api/users/import` 已是真实接口
- 但处理方式仍是逐条校验、逐条创建

影响：

- 大批量导入时性能和错误反馈粒度仍有限

#### P2-3 后台壳层接口与真实能力混杂

表现：

- `positions` 是空壳接口
- Team 协作仍主要是界面包装

影响：

- 新成员或新文档容易把“有入口”误判为“已交付”

### 3.3 P3 风险

#### P3-1 历史文档和当前实现长期脱节

表现：

- 仓库里仍保留不少旧方案和旧结论
- 例如旧版文档还在描述 MongoDB / ClickHouse 主线

影响：

- 开发、测试和产品判断容易被错误前提带偏

#### P3-2 前端仍有一批占位页面和预留功能

表现：

- 存在少量“页面占位”视图
- 结果页和分享页也有部分保留按钮/占位说明

影响：

- 功能边界不够清晰，验收时需要格外注意“真功能”和“预留 UI”的区分

## 4. 当前总体判断

当前项目已经具备以下真实交付能力：

- 问卷主链路闭环
- 题型主链路闭环
- 上传题闭环
- 答卷管理与导出
- 用户、角色、部门、文件夹、消息、审计等后台基础能力

但它还不应被判断为“企业协作与高级分析全部完成”，主要差距集中在：

- 题型体系内部仍需继续收口
- 独立 Team 领域缺失
- 分析可视化仍有明显占位部分
- 部分后台接口仍是壳层

## 5. 建议的后续优先级

1. 继续收口题型体系，把共享注册表变成唯一事实源。
2. 把结果页中仍是占位的趋势图、地域图、系统图真正接上数据。
3. 明确 Team 是否需要独立后端模型；如果需要，先定义边界再开发。
4. 将 `positions`、附件打包下载等壳层能力要么补齐，要么在产品层显式下线。
5. 将数据库 schema 迁移从“启动时补齐”逐步演进到版本化迁移。
