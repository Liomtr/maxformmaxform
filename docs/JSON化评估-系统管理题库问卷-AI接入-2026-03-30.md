# AI 接入现状重评估：系统管理、题库、问卷参数与问卷逻辑

更新时间：2026-03-31

## 1. 评估范围

本次重新评估基于当前仓库真实代码，重点回答四个问题：

- 系统管理是否已经具备受控的 AI 执行动作协议
- 题库是否已经具备可供 AI 使用的结构化内容承载能力
- 问卷参数是否已经形成稳定的 JSON 输入输出边界
- 问卷逻辑是否已经具备共享类型、后端校验和 dry-run 能力

本次结论以当前代码为准，不沿用旧版本文档中的历史假设。

## 2. 总体结论

当前系统的 AI 接入状态，不应再概括为“准备阶段”。

更准确的判断是：

- 系统管理 AI：已经落地第一版正式接入
- 问卷 JSON AI：已经具备受控导入和 dry-run 校验能力
- 问卷逻辑 AI：已经具备共享类型、清洗和后端结构校验
- 题库 AI：已经有结构化内容容器，但还没有收口成统一强约束 DSL

因此，当前系统最适合推进的不是“重新讨论要不要接 AI”，而是分层推进：

1. 继续做实管理域 AI 动作协议
2. 将问卷 JSON 导入升级为真正的服务端 AI 生成链路
3. 把题库内容容器继续收口为统一题目 DSL

## 3. 重新评估后的分项结论

### 3.1 系统管理 AI：已正式落地，不再只是建议方案

这一块与旧文档相比变化最大。

当前已经存在完整的管理域 AI 协议与执行链路：

- 后端路由：`/api/management-ai/protocol`
- 执行入口：`/api/management-ai/actions`
- 执行账本：`/api/management-ai/executions`
- 账本导出：`/api/management-ai/executions/export`
- 前端入口：`frontend/src/views/admin/Config.vue`

当前管理域 AI 具备以下能力：

- 统一协议版本 `2026-03-31`
- 统一动作 envelope：`management.action`
- `dryRun` 校验
- `idempotencyKey` 幂等执行
- 执行结果回放
- 失败请求回填
- 执行账本查询与导出
- 管理员权限边界
- 通过现有 service 执行，而不是直接写表
- 审计与消息记录

当前已支持的动作类型包括：

- 用户：创建、更新、删除、重置密码
- 角色：创建、更新、删除
- 部门：创建、更新、删除
- 岗位：创建、更新、删除
- 流程：创建、更新、删除
- 题库仓库：创建、更新、删除
- 题库题目：创建、删除

结论：

**系统管理 AI 已经从“可设计”进入“已接入、可执行、带审计边界”的阶段。**

### 3.2 问卷参数 JSON：已经成熟，可作为 AI 的标准输入输出

这部分依然是当前最成熟的 AI 接入基础。

当前问卷参数具备以下事实：

- 前后端共享题型映射与题目 DTO
- 编辑器保存时输出的是标准问卷 JSON
- 后端创建、更新、发布前都会走统一标准化和校验
- 前端存在 `validate` 与 `dry-run` 接口

现有能力包括：

- 问卷草稿校验：`POST /api/surveys/validate`
- 问卷 JSON dry-run：`POST /api/surveys/dry-run`
- 问卷字段清洗：共享 `survey.contract`
- 题型标准化与提交校验：`backend/src/utils/questionSchema.js`

当前已经明确结构化的问卷字段包括：

- `type / uiType`
- `title / titleHtml / description`
- `required / order / hideSystemNumber`
- `options`
- `matrix`
- `validation`
- `upload`
- `logic.visibleWhen`
- `jumpLogic`
- `optionGroups`
- `quotasEnabled / quotaMode / quotaShowRemaining / quotaFullText`
- `autoSelectOnAppear`
- `examConfig`

结论：

**问卷参数层已经具备作为 AI 标准 JSON 协议的条件，适合继续加强而不是推倒重来。**

### 3.3 问卷逻辑：不再是“类型松散、后端不校验”的状态

旧文档在这一点上已经过期。

当前问卷逻辑已经具备三层保护：

- 共享类型层：`shared/survey.contract.d.ts`
- 写入清洗层：`shared/survey.contract.js`
- 后端结构校验层：`backend/src/utils/questionSchema.js`

当前已经被显式建模或校验的逻辑结构包括：

- `logic.visibleWhen`
- 选项级 `visibleWhen`
- `jumpLogic.byOption`
- `jumpLogic.unconditional`
- `optionGroups`
- `quotaMode`
- `autoSelectOnAppear`

后端当前已校验的关键约束包括：

- `visibleWhen` 必须引用前置题目
- 操作符必须在受支持集合内
- 依赖选项值必须真实存在
- 跳题目标必须跳向后续题目或 `end`
- `optionGroups` 必须合法且不能重叠
- 配额和自动选择等能力只能用于支持的题型

结论：

**问卷逻辑已经具备“可供 AI 安全生成后再由服务端校验”的基础条件。**

仍然需要注意的是：

- 逻辑目前主要校验结构合法性，不是语义最优性
- 复杂逻辑组合仍需 dry-run 和人工复核

### 3.4 题库：已经有结构化内容容器，但还没有形成统一强 DSL

这一部分也需要修正文档结论。

旧文档认为题库只有：

- `title`
- `type`
- `difficulty`
- `score`

这个判断已经不准确。

当前题库除了基础列，还支持通过 `content` 承载结构化内容，并在 DTO 中对外展开为：

- `stem`
- `options`
- `correctAnswer`
- `analysis`
- `tags`
- `knowledgePoints`
- `applicableScenes`
- `aiMeta`
- `surveyQuestion`

其中：

- `surveyQuestion` 可以承载与问卷题接近的结构，用于题库题导入编辑器
- `aiMeta` 可以记录 AI 生成来源与审核状态
- 前端已支持从问卷导出当前题目回题库，并附带标签、知识点、场景和 AI 元数据

这说明题库已经不是“纯轻量索引目录”，而是进入了“结构化内容容器”的阶段。

但它仍然没有达到问卷 DSL 的成熟度，主要问题在于：

- `content` 允许承载较多自由字段
- `surveyQuestion` 当前只要求是对象，还不是强类型协议
- 缺少与问卷题相同级别的标准化和完整后端约束
- 缺少题库题的版本、审核、可发布状态等治理字段

结论：

**题库已经具备 AI 可用的结构化基础，但仍处于“半结构化内容容器”阶段，不宜宣称为成熟题目 DSL。**

## 4. 当前实际可用的 AI 接入方式

### 4.1 管理域 AI 接入

当前已经可以：

- 获取协议
- 生成受控动作 JSON
- 先 `dryRun`
- 再携带 `idempotencyKey` 执行
- 查询执行账本
- 导出账本
- 回填失败请求重新执行

适合当前落地的场景：

- 批量管理动作的 AI 编排
- 管理后台的 AI 助手
- 审批前的 AI 方案生成与 dry-run 校验

### 4.2 问卷 AI 接入

当前存在两条链路：

- 编辑器中的 AI JSON 导入
- 服务端 `validate` / `dry-run`

需要明确边界：

- 编辑器中的“AI 助手”本质上是前端 JSON 解析导入，不是后端 LLM 接口
- 当前系统已经允许把 AI 产出的问卷 JSON 送入服务端做 dry-run 校验
- 因此，真正缺的不是问卷 JSON 协议，而是“模型调用层”和“生成任务编排层”

### 4.3 题库 AI 接入

当前已经可以：

- 让 AI 生成题干、选项、标签、知识点、场景等结构化字段
- 将问卷题导出为题库题
- 从题库题回灌到问卷编辑器

当前还不适合：

- 直接让 AI 大规模自动沉淀题库且不经审核
- 把题库当成完全统一、强约束、可直接复用到任意题型的标准 DSL

## 5. 当前不应继续使用的旧判断

以下说法已不再准确：

- “系统管理还只是适合做 JSON 指令层，尚未真正接入 AI”
- “问卷逻辑类型约束和后端校验不够强”
- “题库题目目前只有 title/type/difficulty/score 四个轻量字段”

更准确的替代说法应为：

- 系统管理 AI 指令层已经落地并具备执行闭环
- 问卷逻辑已经有共享 contract、清洗和服务端结构校验
- 题库已具备结构化内容容器，但尚未升级为统一强 DSL

## 6. 下一步建议

### 6.1 第一优先级：把问卷 AI 从“前端导入”升级为“服务端正式链路”

建议新增：

- 服务端问卷生成接口
- Prompt 模板与输出协议版本
- 与 `POST /api/surveys/dry-run` 联动的自动校验
- 生成结果审计字段

目标：

AI 直接产出标准问卷 JSON，服务端完成校验、审计、落库前确认。

### 6.2 第二优先级：继续增强管理域 AI

建议继续补：

- 批量动作协议
- 多步骤事务编排
- 更细粒度失败分类
- 执行账本的筛选与可视化
- 动作级权限矩阵

目标：

把当前单动作、强受控的管理域 AI，升级为企业可用的 AI 管理助手底座。

### 6.3 第三优先级：将题库收口为统一 DSL

建议补：

- `surveyQuestion` 的明确共享类型
- 题库题标准化与校验器
- 版本号、审核状态、来源信息
- 从题库题到问卷题的稳定映射协议

目标：

让题库不只是“能装内容”，而是“能稳定驱动 AI 出题、组卷、复用和审核”。

## 7. 最终判断

如果目标是“全面接入 AI 辅助管理、出题和组卷”，当前系统已经不再是 2026-03-30 文档所描述的状态。

截至 2026-03-31，更准确的总体判断是：

- 管理域 AI：已落地第一版正式协议和执行闭环
- 问卷 JSON AI：已具备成熟的标准输入输出与 dry-run 校验基础
- 问卷逻辑 AI：已具备共享类型和后端结构校验
- 题库 AI：已进入结构化内容阶段，但仍需继续收口

所以当前最合理的推进方式是：

**以“管理域 AI 执行协议 + 问卷标准 JSON + 题库 DSL 收口”作为三条主线继续推进，而不是重新讨论是否具备 AI 接入基础。**
