# 题型现状评估与丰富题型阶段建议

基于源码状态更新时间：2026-03-25

本文仅以当前仓库实际代码为准，评估题型是否已经形成“创建、保存、预览、填写、提交、结果统计”的闭环，不再沿用 2026-03-24 之前的旧判断。

## 1. 本次复核结论

截至 2026-03-25，项目的题型能力已经从“基础题可用”推进到“中等题型也具备真实闭环”，与旧文档相比需要做出以下修正：

- 编辑器、填写页、提交校验、结果统计已经覆盖：
  - `input`
  - `textarea`
  - `radio`
  - `checkbox`
  - `dropdown(uiType=7)`
  - `slider`
  - `ranking`
  - `upload`
  - `date`
  - `matrix(uiType=20)`
  - `rating(uiType=29)`
  - `scale(uiType=30/34)`
- `stage_explain(uiType=18)` 作为说明型内容块已形成“创建、保存、渲染”闭环，但不属于作答题型。
- `/api/surveys/:id/results` 已经不再只是汇总总数，后端会返回：
  - `questionStats`
  - `systemStats`
  - 完成率、平均时长、今日新增等摘要指标
- 前端答卷分析面板也已消费真实题目统计数据，不再完全依赖 mock。

新的判断应当是：

**项目已进入“题型体系收口与体验补强阶段”，而不是“先把更多题型入口铺满”的阶段。**

## 2. 当前真实题型能力边界

### 2.1 单一事实源

题型的共享注册表位于：

- `shared/questionTypeRegistry.js`

前端和后端分别通过以下文件复用：

- `frontend/src/utils/questionTypeRegistry.ts`
- `backend/src/constants/questionTypes.js`
- `backend/src/utils/questionSchema.js`

这意味着：

- legacy `uiType`
- server `type`
- 是否实现
- 是否需要选项
- 默认选项
- 特殊渲染类型

已经开始集中收口，不应再在各页面各写一套独立枚举。

### 2.2 当前共享注册表中已实现的 legacy 题型

`shared/questionTypeRegistry.js` 中 `implemented=true` 的题型为：

| legacy uiType | server type | 说明 |
|---|---|---|
| `1` | `input` | 填空题 |
| `2` | `textarea` | 简答题 |
| `3` | `radio` | 单选题 |
| `4` | `checkbox` | 多选题 |
| `7` | `radio` | 下拉题，渲染时按 `dropdown` 处理 |
| `8` | `slider` | 滑动条题 |
| `11` | `ranking` | 排序题 |
| `13` | `upload` | 文件上传 |
| `14` | `date` | 日期题 |
| `18` | `input` | 段落说明，渲染为 `stage_explain` |
| `20` | `matrix` | 矩阵单选 |
| `29` | `rating` | 星亮题 |
| `30` | `scale` | NPS 量表 |
| `34` | `scale` | 评价题，作为量表兼容别名 |

未实现但仍保留入口/枚举的包括：

- `21` 矩阵多选
- `24` 矩阵下拉
- `31` 评分单选
- `32` 评分多选
- `36` 比重题

## 3. 各环节闭环评估

### 3.1 编辑器

编辑器主入口位于：

- `frontend/src/views/survey/CreateSurveyPage.vue`

当前已确认：

- 基础题型和中等复杂度题型均可创建并保存
- `20/29/30` 在左侧题型面板有真实入口
- `34` 虽不是主入口题型，但已有兼容分支
- `matrix` 支持行维度编辑
- `rating` 支持评分范围配置
- `scale` 支持量表范围与标签配置
- `upload` 支持数量、大小、后缀限制配置

需要注意：

- 页面顶部注释仍残留一批“UI 占位，未持久化”的旧说明，其中对 `20/29/30/34` 的描述已经过期
- 编辑器内部仍有较多 legacy 编码判断，维护成本偏高

### 3.2 预览与填写页

填写页位于：

- `frontend/src/views/survey/FillSurveyPage.vue`
- `frontend/src/views/survey/FillSurveyMobilePage.vue`

当前已存在真实渲染分支的类型：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `dropdown`
- `date`
- `slider`
- `ranking`
- `upload`
- `rating`
- `scale`
- `matrix`
- `stage_explain`

补充说明：

- 移动端当前复用 PC 填写逻辑，因此题型支持范围一致。
- `matrix` 目前落地的是矩阵单选模式。
- `upload` 走“先上传文件，再提交答卷引用”的模式，不是直接把二进制塞进答卷。

### 3.3 提交校验

提交流程关键逻辑位于：

- `backend/src/utils/questionSchema.js`
- `backend/src/routes/surveys.js`

当前校验能力已经覆盖：

- 发布前标题/题目/选项结构校验
- `radio / checkbox / ranking` 选项合法性校验
- `slider / rating / scale` 数值校验
- `matrix` 行列结构与作答对象校验
- `upload` 文件数量、大小、类型校验
- 必填校验
- 截止时间校验
- 重复提交限制

上传题额外校验：

- 文件是否属于当前问卷
- 文件是否属于当前题号
- `submissionToken` 是否一致
- `uploadToken` 是否匹配

这意味着项目已经不再处于“答卷原样写入 JSON”的阶段。

### 3.4 结果统计

结果统计主逻辑位于：

- `backend/src/routes/surveys.js` 的 `buildQuestionStats`
- `frontend/src/views/survey/SurveyAnswersPanel.vue`

后端当前已生成真实统计的题型包括：

- `radio`
- `checkbox`
- `input`
- `textarea`
- `slider`
- `rating`
- `scale`
- `matrix`
- `ranking`
- `upload`
- `date`

对应统计能力如下：

| 题型 | 当前统计输出 |
|---|---|
| `radio / checkbox` | 选项人数、比例 |
| `input / textarea` | 样本答案 |
| `slider` | 平均值、最小值、最大值 |
| `rating / scale` | 平均分、分布 |
| `matrix` | 行维度下的选项分布 |
| `ranking` | 出现次数、比例、平均排名 |
| `upload` | 文件总数、文件样本 |
| `date` | 样本日期、最早/最晚日期 |

同时还会返回：

- `systemStats.devices`
- `systemStats.browsers`
- `systemStats.operatingSystems`

## 5. 仍然存在的题型相关缺口

虽然题型本身闭环已明显增强，但仍存在三类缺口。

### 5.1 注册表与页面实现仍未完全收口

表现：

- 共享注册表已经建立，但 `CreateSurveyPage.vue`、映射器、统计面板中仍保留不少分散判断
- 页面顶部注释和部分历史兼容代码与真实实现不同步

影响：

- 后续继续扩展题型时，容易出现“一个地方放开、另一个地方没跟上”的问题

### 5.2 结果页里仍有“统计真、图表假”的混合状态

表现：

- 题目统计数据已经是真实接口返回
- 但趋势图、地域分布、系统图表主体仍是占位 UI

影响：

- 容易让人误判为“分析模块已全部完成”

### 5.3 高级题型族谱尚未系统化

表现：

- 目前真正闭环的是 `matrix single / rating / scale`
- `matrix multi / matrix dropdown / ratio` 等仍未交付

影响：

- 如果不先抽象题型家族模型，继续逐个加入口会放大复杂度

## 6. 是否适合继续进入“丰富题型阶段”

结论：

**不适合把下一阶段定义为“全面铺开更多题型入口”。**

更合理的命名应为：

**题型体系收口与分析体验补强阶段**

原因：

- 当前主流题型闭环已经不少，瓶颈不再是“题型数量太少”
- 真正的短板在于统一建模、可视化落地和未完成题型家族的体系化扩展
- 继续盲目增加入口，只会让 legacy 映射更重、维护成本更高

## 7. 下一阶段建议

### 7.1 第一优先级：统一题型注册中心

建议继续把以下能力围绕共享注册表集中：

- 标签与图标
- legacy/server 映射
- 是否需要选项
- 默认配置
- 编辑器配置面板
- 填写页渲染
- 提交校验
- 结果统计适配

目标不是再造一层抽象，而是把当前已经分散在多个文件里的事实源收回到同一套配置。

### 7.2 第二优先级：把真实统计数据渲染成真实图表

优先补齐：

- 收集趋势图
- 地域分布
- 系统环境图表
- 附件批量下载

这一步做完，统计模块的“闭环感”会比再多加 3 个题型更强。

### 7.3 第三优先级：按题型家族扩展

推荐顺序：

1. `matrix multi`
2. `matrix dropdown`
3. `ratio`

原因：

- 这三类是当前能力体系自然延伸
- 复用已有 `matrix / scale / option` 结构的可能性高
- 比继续增加零散异构题型更容易维护

## 8. 总体判断

当前题型系统的真实状态可以概括为：

- 基础题型已稳定闭环
- `rating / scale / matrix / upload` 也已进入真实可用范围
- 统计接口已补到题目级，不再只是汇总总数
- 剩余重点从“能不能作答”转向“能不能长期维护、能不能把结果看得更清楚”

因此，下一阶段最合理的目标不是：

**继续无差别丰富题型数量**

而是：

**把现有题型体系做成一套更统一、更可维护、更可视化的交付能力**
