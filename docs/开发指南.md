# TrustForm 开发指南

基于源码状态更新时间：2026-03-25

本文仅以当前仓库实际代码为准，适用于 `maxformmaxform` 这一套前后端工程，不再沿用旧版文档中的 MongoDB / ClickHouse 双库方案描述。

## 1. 当前技术栈

- 前端：Vue 3 + TypeScript + Vite + Element Plus + Pinia + Vue Router
- 后端：Node.js 18+ + Express 4 + Knex 3
- 数据库：MySQL
- 文件上传：Multer，本地磁盘存储，后端通过 `/uploads` 提供静态访问
- 鉴权：JWT
- 导出：ExcelJS
- 测试：
  - 后端：Node.js 内置 `node:test`
  - 前端：`vue-tsc` 构建校验、Playwright 预留、系统烟测脚本

## 2. 仓库结构

```text
maxformmaxform/
├─ backend/                 # Express + Knex + MySQL
│  ├─ src/config/           # 配置
│  ├─ src/db/               # knex 连接、建表、种子
│  ├─ src/models/           # 数据访问模型
│  ├─ src/routes/           # API 路由
│  ├─ src/utils/            # 题型校验、上传存储
│  └─ test/                 # 路由级测试
├─ frontend/                # Vue 3 前端
│  ├─ src/api/              # API 封装
│  ├─ src/views/            # 页面
│  ├─ src/utils/            # 前端题型注册、上传辅助
│  └─ src/types/            # TS 类型
├─ shared/                  # 前后端共享题型注册表
├─ scripts/                 # 系统烟测脚本
└─ docs/                    # 项目文档
```

## 3. 环境要求

- Node.js 18 或更高版本
- npm 9 或更高版本
- MySQL 8.x

建议本地保留以下端口：

- 前端开发服务：`63000`
- 后端 API：`63002`
- MySQL：默认按 `.env.example` 使用 `3309`

## 4. 环境变量

后端读取 `backend/.env`，当前示例变量如下：

```env
DB_HOST=127.0.0.1
DB_PORT=3309
DB_USER=root
DB_PASSWORD=123456
DB_NAME=survey_system

JWT_SECRET=please-change-me
JWT_EXPIRES_IN=7d

FRONTEND_URL=http://127.0.0.1:63000
PORT=63002
```

补充说明：

- 上传大小上限由 `backend/src/config/index.js` 控制，当前默认单文件 10MB。
- `UPLOAD_PENDING_TTL_HOURS` 可选，控制公开上传但尚未提交答卷的临时文件清理窗口，默认 24 小时。
- 生产环境必须替换 `JWT_SECRET`。

## 7. 数据模型概览

当前 `backend/src/db/migrate.js` 会确保以下核心表存在：

- `roles`
- `depts`
- `users`
- `surveys`
- `answers`
- `files`
- `folders`
- `messages`
- `audit_logs`

其中：

- `surveys.questions` 为 JSON 字段，存储问卷题目定义
- `answers.answers_data` 为 JSON 字段，存储答卷内容
- `files` 同时承担普通文件与问卷上传附件的记录
- `surveys` 支持 `folder_id`、软删除字段 `deleted_at` / `deleted_by`

## 8. 题型体系

### 8.1 单一事实源

前后端共享题型注册表位于：

- `shared/questionTypeRegistry.js`

这里统一维护：

- legacy `uiType`
- server `type`
- 是否已实现
- 是否需要选项
- 默认选项
- 特殊渲染类型

### 8.2 当前服务端支持的题型

- `input`
- `textarea`
- `radio`
- `checkbox`
- `date`
- `slider`
- `ranking`
- `upload`
- `rating`
- `scale`
- `matrix`

### 8.3 当前已标记为 implemented 的 legacy 题型

- `1` 填空题
- `2` 简答题
- `3` 单选题
- `4` 多选题
- `7` 下拉题
- `8` 滑动条题
- `11` 排序题
- `13` 文件上传
- `14` 日期
- `18` 段落说明
- `20` 矩阵单选
- `29` 星亮题
- `30` NPS 量表
- `34` 评价题

说明：

- `7` 通过 `uiType=7` 渲染为下拉，但服务端按 `radio` 存储。
- `18` 是说明型内容块，不参与答题统计。
- `34` 兼容映射到 `scale`，更偏历史别名。

## 9. 问卷与答卷主链路

### 9.1 问卷

核心路由在 `backend/src/routes/surveys.js`：

- 创建、更新、删除、发布、关闭
- 回收站、恢复、彻底删除
- 文件夹移动
- 公开读取已发布问卷
- 提交答卷
- 查询结果统计

### 9.2 答卷

核心路由在 `backend/src/routes/answers.js`：

- 分页查询答卷
- 查询答卷数量
- 查看单份答卷
- 批量删除
- 导出问卷答卷 Excel

### 9.3 提交校验

`backend/src/utils/questionSchema.js` 当前已覆盖：

- 发布前题目合法性校验
- 必填校验
- 单选/多选/排序选项合法性校验
- `slider / rating / scale` 数值校验
- `matrix` 行列结构校验
- `upload` 文件数量、大小、类型校验

## 10. 文件上传机制

上传相关实现位于：

- `backend/src/routes/files.js`
- `backend/src/routes/surveys.js`
- `backend/src/utils/uploadStorage.js`
- `backend/src/models/File.js`

当前分两类：

### 10.1 管理后台普通上传

- `POST /api/files/upload`
- `POST /api/files/upload/image`

要求登录。

### 10.2 问卷公开上传

- `POST /api/surveys/:id/uploads`

特点：

- 允许已发布问卷的公开附件上传
- 可绑定 `questionId + submissionToken`
- 返回 `uploadToken`
- 真正提交答卷时再由后端二次校验并绑定到 `answers`
- 未提交的临时附件会按 TTL 清理

## 13. 常见开发约定

### 13.1 题型改动

新增或调整题型时，至少同步检查：

- `shared/questionTypeRegistry.js`
- `backend/src/utils/questionSchema.js`
- `frontend/src/utils/questionTypeRegistry.ts`
- `frontend/src/views/survey/CreateSurveyPage.vue`
- `frontend/src/views/survey/FillSurveyPage.vue`
- `frontend/src/views/survey/SurveyAnswersPanel.vue`

### 13.2 结果统计改动

题目级统计逻辑当前集中在：

- `backend/src/routes/surveys.js` 中的 `buildQuestionStats`
- `frontend/src/views/survey/SurveyAnswersPanel.vue`

### 13.3 不要再按旧架构开发

以下认知已经过期，不应继续作为开发前提：

- MongoDB 是主库
- ClickHouse 承担结果统计
- 统计接口只有总提交数
- 题型只支持基础 4 类

## 14. 当前已知缺口

- 结果页中的趋势图、地域分布、系统图表仍有前端占位内容
- 附件批量下载尚未有后端接口
- 高级分析 / SPSS 工具仍是界面预留
- `positions` 仍是空壳接口
- 协作能力当前主要复用 `部门 + 成员`，独立 Team 领域尚未建立

## 15. 建议的开发顺序

1. 继续把题型注册、提交校验、结果统计收口到共享模型。
2. 补齐结果页中仍是占位的图表与附件下载。
3. 再决定是否扩展矩阵多选、矩阵下拉、比重题等更复杂题型。
4. 若要做企业协作，再单独抽象 Team / Invite / Approval，而不是继续用部门语义硬扛。
