# 系统结构评估与优化建议
更新时间：2026-03-28

## 1. 评估范围

本次评估只基于当前仓库源码与 2026-03-28 的实测结果，不沿用旧版 MongoDB / ClickHouse 双库口径。

重点检查：

- `backend/` 当前真实启动入口、分层结构、问卷域与答卷域的服务化程度
- `frontend/src/views/survey/` 与 `frontend/src/composables/editor/` 的拆分现状
- `shared/` 是否已承担题型语义、提交契约、统计口径的统一职责
- 系统验证入口是否仍能覆盖当前主链路

## 2. 本次结论

当前系统已经进入“可持续迭代的模块化单体”阶段，不再适合描述为“巨型页面 + 巨型路由”的原型工程。

更准确的判断是：

- 后端主线已经稳定在 `Express + Knex + MySQL`。
- 问卷相关链路已经形成 `route + policy + service + repository + transaction` 的稳定分层。
- 答卷链路也已经完成一轮服务化，查询、删除、导出分别进入 `answerQueryService`、`answerCommandService`、`answerExportService`。
- 结果统计不再只是临时聚合，已经引入 `SurveyResultsSnapshot` 做快照缓存。
- 前端编辑器已完成页面壳拆分、领域 composable 拆分，并把 `editorContext` 对外边界收敛为显式 contract。
- 管理域已经开始按统一契约推进，`shared/management.contract.js`、`deptService`、`roleService`、`positionService`、`auditService` 等分层节点已经落地。
- 系统联调入口仍可真实跑通，而且今天已经按最新代码验证通过。

一句话结论：

**当前重点不再是推翻式重构，而是把现有分层模式推广到更多模块，并继续补齐共享契约与回归验证。**

## 3. 当前真实结构

### 3.1 仓库级结构

```text
maxformmaxform/
├─ backend/     后端服务
├─ frontend/    前端应用
├─ shared/      共享题型与契约
├─ scripts/     系统冒烟与辅助脚本
└─ docs/        项目文档
```

### 3.2 后端结构

当前后端真实入口：

- `backend/server.js`
- `backend/app.js`

问卷与答卷域的主链路已经基本形成如下关系：

```text
HTTP Request
  -> Route
    -> Policy / Access Check
      -> Service
        -> Repository + Transaction
          -> Model
            -> JSON / Download Response
```

当前可确认的后端分层节点：

- 路由
  - `backend/src/routes/surveys.js`
  - `backend/src/routes/answers.js`
  - `backend/src/routes/files.js`
- 策略
  - `backend/src/policies/surveyPolicy.js`
  - `backend/src/policies/accessPolicy.js`
- 服务
  - `backend/src/services/surveyQueryService.js`
  - `backend/src/services/surveyCommandService.js`
  - `backend/src/services/surveyUploadService.js`
  - `backend/src/services/surveyResultsService.js`
  - `backend/src/services/answerQueryService.js`
  - `backend/src/services/answerCommandService.js`
  - `backend/src/services/answerExportService.js`
  - `backend/src/services/fileQueryService.js`
  - `backend/src/services/fileCommandService.js`
- 仓储与事务
  - `backend/src/repositories/surveyAggregateRepository.js`
  - `backend/src/db/transaction.js`
- 共享校验与契约
  - `backend/src/utils/questionSchema.js`
  - `shared/survey.contract.js`
  - `shared/pagination.contract.js`

补充判断：

- `surveyService.js` 现在更像 barrel 导出层，不再承载核心业务。
- `answers.js` 相比上一轮评估已经明显收缩，但下载响应装配仍留在 route 层。
- 部门、角色、职位、消息、审计等后台模块已经进入 `route -> service -> repository/contract` 迁移阶段，但访问控制、事务边界与聚合复杂度仍未完全对齐问卷域风格。

### 3.3 前端结构

问卷前端当前不是单页巨石，而是“页面壳 + 子面板 + 领域 composable”。

编辑器页面层：

- `frontend/src/views/survey/CreateSurveyPage.vue`
- `frontend/src/views/survey/CreateSurveyPageShell.vue`
- `frontend/src/views/survey/CreateSurveyQuestionConfigPanel.vue`
- `frontend/src/views/survey/CreateSurveyQuestionListPanel.vue`
- `frontend/src/views/survey/CreateSurveyLogicSettings.vue`
- `frontend/src/views/survey/CreateSurveyPublishPanel.vue`

编辑器编排层：

- `frontend/src/composables/useSurveyEditor.ts`
- `frontend/src/composables/editor/editor-core.ts`
- `frontend/src/composables/editor/editor-logic.ts`
- `frontend/src/composables/editor/editor-batch.ts`
- `frontend/src/composables/editor/editor-quota.ts`
- `frontend/src/composables/editor/editor-richtext.ts`

填写页逻辑也已经按能力拆开：

- `frontend/src/views/survey/useFillSurveyVisibility.ts`
- `frontend/src/views/survey/useFillSurveyJumpLogic.ts`
- `frontend/src/views/survey/useFillSurveyQuota.ts`
- `frontend/src/views/survey/useFillSurveyUpload.ts`

当前前端的关键进展：

- `useSurveyEditor.ts` 已从“业务巨石”收缩为装配层。
- 页面层 contract 已显式化，`editorContext` 不再通过 `ReturnType` 隐式外溢。
- 编辑器拆分重点已从“是否拆文件”转到“是否继续缩小子面板依赖面”。

### 3.4 共享层结构

`shared/` 已经从“常量目录”提升为问卷域共享事实源：

- `shared/questionTypeRegistry.js`
  - 统一题型定义
  - 兼容 legacy `uiType`
  - 统一提交语义与上传能力
- `shared/questionModel.js`
  - 统一题目级统计口径
- `shared/survey.contract.js`
  - 统一问卷状态、错误码、上传/提交 DTO
- `shared/pagination.contract.js`
  - 统一分页归一化与返回结构

这意味着当前系统已经不再依赖“前后端各维护一套题型语义”的旧模式。

## 4. 已确认的结构进展

### 4.1 后端服务化范围继续扩大

相比上一轮评估，本轮最重要的变化不是“多了几个文件”，而是服务边界更清晰：

- 问卷读取、写入、上传、结果已经完全分开。
- 答卷查询、批量删除、Excel 导出、附件打包也已经拆成独立服务。
- 文件读写也开始进入 `query/command` 分离。
- 后端测试入口已经不再只覆盖路由，而是覆盖 route + service 两层。

### 4.2 结果统计进入“实时聚合 + 快照缓存”阶段

`backend/src/services/surveyResultsService.js` 当前已经引入 `SurveyResultsSnapshot`。

这说明：

- 系统仍然保留 JSON 写模型的题型迭代灵活性。
- 同时开始用轻量快照降低重复统计成本。
- 当前还没有形成独立读模型，但已经不是“每次都完全裸算”的最早阶段。

### 4.3 前端编辑器边界更稳定

本轮已确认：

- 编辑器 UI 壳和业务编排已经解耦。
- `useSurveyEditor.ts` 暴露的是显式 contract，而不是整块内部状态。
- `CreateSurveyPageShell` 与子面板之间的类型边界更稳定，后续继续拆分时更不容易牵一发动全身。

## 5. 当前仍存在的主要问题

### P1. 后端分层风格尚未全局统一

问卷、答卷、文件模块已经明显向服务化靠拢，后台管理域也已开始补齐 service/repository/contract，但当前仍存在两个不一致点：

- 不同管理模块的访问控制边界不统一，有的仍主要依赖 route 层守卫
- 聚合读写与事务边界尚未像问卷域那样系统化

这不是阻塞问题，但会持续拉低整体一致性。

### P1. 共享契约仍然主要集中在问卷域

`shared/` 现在已经覆盖题型、分页、问卷错误码，但仍未形成更完整的跨域共享层，例如：

- 用户/部门/文件 DTO
- 通用错误码枚举
- 更统一的列表返回契约

### P1. 系统级验证已补上浏览器 E2E，但覆盖面仍偏窄

当前验证矩阵已经包含：

- 后端 `node:test`
- 系统级 API 冒烟
- 前端类型检查 + 构建
- Playwright 浏览器 E2E

但还缺：

- 管理后台、上传题、权限边界等更多浏览器回归
- 编辑器/填写页/结果页的可视化回归

### P2. 结果快照已出现，但运维可观测性仍不足

目前已经有快照缓存与失效逻辑，但缺少：

- 快照命中率观察
- 快照重建耗时追踪
- 结果接口在大样本下的基线指标

### P2. 旧文档漂移仍然存在

仓库里仍有部分文档带有历史“双库架构”痕迹，需要继续清理，避免读者先看到旧口径再回头纠偏。

## 6. 2026-03-28 实测结论

本轮不是纯静态阅读，已完成以下实测：

- `cmd /c npm test` 于 `backend/` 通过，`66 / 66`
- `cmd /c npm run test:e2e` 于 `frontend/` 通过，`3 / 3`
- `node scripts/system-smoke.mjs` 于仓库根目录通过，`64 / 64`
- `cmd /c npm run smoke:system` 于 `frontend/` 通过，`64 / 64`
- `cmd /c npm run build` 于 `frontend/` 通过

本轮系统冒烟已经验证的关键链路包括：

- 登录与注册
- 部门、职位、成员导入
- 文件夹与删除保护
- 问卷创建、发布、分享、移动目录、回收站、恢复、关闭
- 上传题公开上传、提交校验、跨题绑定保护
- 公开答卷提交
- 答卷列表、计数、详情、批量删除
- 结果统计、系统统计、附件/Excel 导出
- 消息、审计与清理回收

同时，本轮冒烟脚本已经同步修复三个现实问题：

- `runId` 从秒级时间戳改为 `Date.now() + pid + random`，消除并发入口冲突
- 脚本预热逻辑从旧的 `seed()` 调整为当前实际存在的 `ensureBaseRoles()`
- 审计日志读取断言改为匹配分页契约，避免把正常分页响应误判为失败

## 7. 下一步建议

第一优先级：

- 继续把 `route -> model` 风格的后台模块向 `policy + service + repository` 迁移
- 把共享契约从问卷域扩展到更多管理域
- 为结果快照补最小可观测性

第二优先级：

- 扩大浏览器级 E2E 覆盖面，补到管理后台、上传题和权限场景
- 为高价值页面补轻量视觉回归或截图基线

第三优先级：

- 持续清理旧文档中的历史架构描述
- 在 README / 开发指南中统一测试与验收口径

## 8. 统一口径

当前系统应统一描述为：

- 后端是 `Express + Knex + MySQL` 的模块化单体。
- 问卷域已经形成稳定服务分层，答卷/文件域正在补齐同样模式。
- 前端问卷编辑器已经完成页面层与 composable 层双轮拆分，边界趋于稳定。
- `shared/` 已成为题型语义、问卷契约与统计口径的统一事实源。
- 当前主要任务是扩充分层、共享契约和回归验证，而不是重新推翻架构。
